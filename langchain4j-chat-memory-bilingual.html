Overview 概述

Maintaining and managing ChatMessages manually is cumbersome. Therefore, LangChain4j offers a ChatMemory abstraction along with multiple out-of-the-box implementations.

EN

ChatMemory can be used as a standalone low-level component, or as a part of a high-level component like AI Services.

ChatMemory acts as a container for ChatMessages (backed by a List), with additional features like:

  • — Eviction policy
  • — Persistence
  • — Special treatment of SystemMessage
  • — Special treatment of tool messages
中文

ChatMemory 既可作为独立的底层组件使用，也可作为 AI Services 等高层组件的一部分。

ChatMemory 充当 ChatMessage 的容器（底层由 List 支持），并具备以下附加功能：

  • — 淘汰策略（Eviction policy）
  • — 持久化（Persistence）
  • — SystemMessage 特殊处理
  • — 工具消息特殊处理
02

Memory vs History 记忆 与 历史

Please note that "memory" and "history" are similar, yet distinct concepts. LangChain4j currently offers only "memory", not "history".

Memory · 记忆

ChatMemory

Keeps some information presented to the LLM to make it behave as if it "remembers" the conversation. Depending on the memory algorithm, it can modify history in various ways:

  • Evict some messages
  • Summarize multiple messages
  • Remove unimportant details
  • Inject extra information (e.g., for RAG)
  • Inject instructions (e.g., for structured outputs)

保留部分信息呈现给 LLM，使其表现得像是"记住"了对话。根据算法不同，可以淘汰、摘要、清洗或注入额外信息。

History · 历史

Chat History

Keeps all messages between the user and AI intact. History is what the user sees in the UI. It represents what was actually said.

If you need to keep an entire history, please do so manually — LangChain4j does not provide built-in history management.

完整保留用户与 AI 之间的所有消息，是用户在 UI 中看到的内容，代表实际发生过的对话。如需完整历史记录，请自行管理。

03

Eviction Policy 淘汰策略

An eviction policy is necessary to fit within the LLM's context window, control cost, and control latency. Currently, LangChain4j offers two out-of-the-box implementations.

EN

Why Eviction?

An eviction policy is necessary for several reasons:

Context window — There is a cap on the number of tokens LLM can process at once. At some point, conversation might exceed this limit.

Cost control — Each token has a cost, making each call to the LLM progressively more expensive. Evicting unnecessary messages reduces the cost.

Latency control — The more tokens are sent to the LLM, the more time it takes to process them.

中文

为什么需要淘汰？

淘汰策略的必要性体现在以下几个方面：

上下文窗口 — LLM 一次能处理的 token 数量有上限，对话可能超出此限制。

成本控制 — 每个 token 都有成本，每次调用 LLM 的费用会逐渐增加。淘汰不必要的消息可降低成本。

延迟控制 — 发送给 LLM 的 token 越多，处理时间就越长。

Simple

MessageWindowChatMemory

Functions as a sliding window, retaining the N most recent messages and evicting older ones that no longer fit. However, because each message can contain a varying number of tokens, this is mostly useful for fast prototyping.

滑动窗口机制，保留最近 N 条消息。由于每条消息 token 数不一致，主要用于快速原型开发。

Advanced

TokenWindowChatMemory

A more sophisticated option that also operates as a sliding window but focuses on keeping the N most recent tokens. Messages are indivisible — if a message doesn't fit, it is evicted completely. Requires a TokenCountEstimator to count tokens in each ChatMessage.

更精细的滑动窗口，按最近 N 个 token 保留。消息不可分割，若放不下则整条淘汰。需要 TokenCountEstimator 来计数。

04

Persistence 持久化

By default, ChatMemory implementations store ChatMessages in memory. If persistence is required, a custom ChatMemoryStore can be implemented.

EN

Custom ChatMemoryStore

If persistence is required, a custom ChatMemoryStore can be implemented to store ChatMessages in any persistent store of your choice. Implement three methods:

getMessages(Object memoryId) — Called whenever the user of the ChatMemory requests all messages. Typically once per LLM interaction.

updateMessages(Object memoryId, List<ChatMessage>) — Called every time a new ChatMessage is added. Usually twice per interaction (UserMessage + AiMessage).

deleteMessages(Object memoryId) — Called when ChatMemory.clear() is called.

中文

自定义 ChatMemoryStore

如需持久化，可实现自定义 ChatMemoryStore，将 ChatMessage 存储到任意持久化存储中。需实现三个方法：

getMessages(Object memoryId) — 每当 ChatMemory 的使用者请求所有消息时调用，通常每次 LLM 交互调用一次。

updateMessages(Object memoryId, List<ChatMessage>) — 每次添加新 ChatMessage 时调用，通常每次交互调用两次（UserMessage + AiMessage）。

deleteMessages(Object memoryId) — 当调用 ChatMemory.clear() 时调用。

05

Special Treatment of SystemMessage SystemMessage 的特殊处理

SystemMessage is a special type of message, so it is treated differently from other message types.

EN

SystemMessage is treated differently from other message types:

Always retained — Once added, a SystemMessage is always retained.

Only one — Only one SystemMessage can be held at a time.

Same content — If a new SystemMessage with the same content is added, it is ignored.

Different content — If a new SystemMessage with different content is added, it replaces the previous one. By default, the new SystemMessage is added to the end of the message list. You can change this by setting alwaysKeepSystemMessageFirst property when creating ChatMemory.

中文

SystemMessage 与其他消息类型的处理方式不同：

始终保留 — 一旦添加，SystemMessage 将始终保留。

仅有一条 — 同一时间只能持有一条 SystemMessage。

相同内容 — 如果添加内容相同的 SystemMessage，则会被忽略。

不同内容 — 如果添加内容不同的 SystemMessage，则会替换之前的那条。默认情况下，新消息被添加到消息列表末尾。可以通过创建 ChatMemory 时设置 alwaysKeepSystemMessageFirst 属性来改变此行为。

06

Special Treatment of Tool Messages 工具消息的特殊处理

If an AiMessage containing ToolExecutionRequests is evicted, the following orphan ToolExecutionResultMessage(s) are also automatically evicted.

EN

If an AiMessage containing ToolExecutionRequests is evicted, the following orphan ToolExecutionResultMessage(s) are also automatically evicted.

This is to avoid problems with some LLM providers (such as OpenAI) that prohibit sending orphan ToolExecutionResultMessage(s) in the request.

中文

如果包含 ToolExecutionRequests 的 AiMessage 被淘汰，则其后孤立的 ToolExecutionResultMessage 也会被自动淘汰。

这是为了避免某些 LLM 提供商（如 OpenAI）禁止在请求中发送孤立的 ToolExecutionResultMessage 而导致问题。
