Tools (Function Calling)

让 LLM 不止于生成文本 —— 当需要时，它可以调用你定义的工具：搜索网页、调用 API、执行代码，将 AI 从对话引擎升级为行动引擎。

@Tool ToolSpecification ToolExecutor ToolProvider ToolSearch MCP
01

概述 / Overview

EN

Some LLMs, in addition to generating text, can also trigger actions. Known as "tools" or "function calling", this allows the LLM to call one or more available tools when necessary. A tool can be anything: a web search, an external API call, or executing specific code. LLMs cannot call tools themselves; they express the intent to call a tool, and we developers execute it and report back the results.

中文

某些 LLM 除了生成文本外，还能触发操作。这被称为"工具"或"函数调用"，允许 LLM 在必要时调用一个或多个可用工具。工具可以是任何东西：网页搜索、外部 API 调用或执行特定代码。LLM 本身无法真正调用工具，而是表达调用意图，由开发者执行并将结果反馈给 LLM。

关键提示：提供清晰明确的工具名称、功能描述以及每个参数的描述，能大幅提高 LLM 正确调用工具的概率。一个好的经验法则：如果人类能理解工具的用途和使用方式，LLM 大概率也能理解。
数学示例：有工具 vs 无工具
WITHOUT TOOLS · 无工具
Request: "What is the square root of 475695037565?"
Response: "approximately 689710." — 接近但不正确。
WITH TOOLS · 有工具
Request → LLM 请求 squareRoot(475695037565)
执行 Math.sqrt() → 回传结果 → Response: "689706.486532." — 精确！
02

两层抽象 / Two Levels of Abstraction

EN

Low-level: Use ChatModel and ToolSpecification APIs directly. You manually create tool specs, handle execution, and manage the request/response cycle.

High-level: Use AI Services with @Tool-annotated Java methods. AI Services automatically convert methods to ToolSpecifications, execute tools when LLM requests, and feed results back.

中文

低级 API：直接使用 ChatModel 和 ToolSpecification。手动创建工具规格、处理执行并管理请求/响应周期。

高级 API：使用 AI Services 配合 @Tool 注解的 Java 方法。AI Services 自动将方法转换为 ToolSpecification，在 LLM 请求时执行工具并将结果反馈。

03

低级工具 API / Low-Level Tool API

创建 ToolSpecification
EN

ToolSpecification contains all tool info: name, description, parameters (with their descriptions), and metadata. Two ways to create: manually via builder, or from annotated classes via ToolSpecifications.toolSpecificationsFrom().

中文

ToolSpecification 包含工具的所有信息：名称、描述、参数（含描述）和元数据。两种创建方式：通过 Builder 手动构建，或通过 ToolSpecifications.toolSpecificationsFrom() 从注解类生成。

Java Manual Creation 
ToolSpecification toolSpecification = ToolSpecification.builder()
    .name("getWeather")
    .description("Returns the weather forecast for a given city")
    .parameters(JsonObjectSchema.builder()
        .addStringProperty("city",
            "The city for which the weather forecast should be returned")
        .addEnumProperty("temperatureUnit",
            List.of("CELSIUS", "FAHRENHEIT"))
        .required("city")
        .build())
    .build();
Java From Annotated Class 
class WeatherTools {
    @Tool("Returns the weather forecast for a given city")
    String getWeather(
        @P("The city for which the weather forecast should be returned")
        String city,
        TemperatureUnit temperatureUnit) { ... }
}

List<ToolSpecification> toolSpecs =
    ToolSpecifications.toolSpecificationsFrom(WeatherTools.class);
使用 ChatModel 调用
Java ChatModel + Tools 
// 1. 发送带工具的请求
ChatRequest request = ChatRequest.builder()
    .messages(UserMessage.from("What will the weather be like in London tomorrow?"))
    .toolSpecifications(toolSpecifications)
    .build();
ChatResponse response = model.chat(request);
AiMessage aiMessage = response.aiMessage();

// 2. 检查 LLM 是否请求调用工具
if (aiMessage.hasToolExecutionRequests()) {
    // 3. 手动执行工具
    String result = "It is expected to rain in London tomorrow.";
    ToolExecutionResultMessage toolResult =
        ToolExecutionResultMessage.from(toolExecutionRequest, result);

    // 4. 将结果发回 LLM
    ChatRequest request2 = ChatRequest.builder()
        .messages(List.of(userMessage, aiMessage, toolResult))
        .toolSpecifications(toolSpecifications)
        .build();
    ChatResponse response2 = model.chat(request2);
}
使用 StreamingChatModel
Java Streaming + Tools 
model.chat(request, new StreamingChatResponseHandler() {
    public void onPartialResponse(String partial) { ... }
    public void onPartialToolCall(PartialToolCall partial) { ... }
    public void onCompleteToolCall(CompleteToolCall complete) { ... }
    public void onCompleteResponse(ChatResponse resp) { ... }
    public void onError(Throwable error) { ... }
});

// 流式 tool call 示例：
// onPartialToolCall(index=0, id="call_abc", name="get_weather", partialArguments="{\"city\":\"London\"}")
// onCompleteToolCall(index=0, id="call_abc", name="get_weather", arguments="{\"city\":\"London\"}")
注意：并非所有 LLM 提供商都支持流式部分工具调用。Bedrock、Google、Mistral、Ollama 等仅返回完整的工具调用，此时 onPartialToolCall 不会被调用。
04

高级工具 API / High-Level Tool API

EN

Annotate any Java method with @Tool and specify it when creating an AI Service. AI Services automatically convert methods to ToolSpecifications, execute the appropriate method when the LLM requests it, and send the return value back to the LLM.

中文

使用 @Tool 注解任意 Java 方法，并在创建 AI Service 时指定。AI Services 自动将方法转换为 ToolSpecification，在 LLM 请求时执行对应方法，并将返回值发回 LLM。

Java @Tool Example 
class Calculator {
    @Tool
    double add(int a, int b) { return a + b; }

    @Tool
    double squareRoot(double x) { return Math.sqrt(x); }
}

MathGenius mathGenius = AiServices.builder(MathGenius.class)
    .chatModel(model)
    .tools(new Calculator())
    .build();

String answer = mathGenius.ask("What is the square root of 475695037565?");
// The square root of 475695037565 is 689706.486532.
@Tool 注解字段：name（工具名称，默认方法名）、value（工具描述）、returnBehavior（返回行为）、metadata（提供商特定元数据 JSON）。
05

工具方法参数 / Tool Method Parameters

EN

@Tool methods can accept: primitives, objects, custom POJOs (with nesting), enums, polymorphic types (sealed interfaces/classes, @JsonSubTypes), List<T>/Set<T>, Map<K,V>, and recursive parameters (OpenAI only). Use @P for parameter name/description, @P(required = false) or Optional<T> for optional params, and @P(defaultValue = "...") for defaults.

中文

@Tool 方法可接受：基本类型、对象、自定义 POJO（可嵌套）、枚举、多态类型（sealed 接口/类、@JsonSubTypes）、List<T>/Set<T>、Map<K,V> 和递归参数（仅 OpenAI）。使用 @P 设置参数名/描述，@P(required = false) 或 Optional<T> 标记可选，@P(defaultValue = "...") 设置默认值。

Java @P · Optional · Default Values 
// @P 自定义参数名和描述
@Tool
void getTemperature(
    @P("Temperature value") double value,
    @P("Unit of temperature") Optional<String> unit) { ... }

// 默认参数值
enum SortBy { RELEVANCE, DATE, RATING }

@Tool
List<Article> searchArticles(
    String query,
    @P(defaultValue = "10") int limit,
    @P(defaultValue = "[\"en\"]") List<String> languages,
    @P(defaultValue = "RELEVANCE") SortBy sortBy) { ... }

// 多态参数 (sealed interface)
sealed interface Animal permits Dog, Cat {}
record Dog(String name, String breed) implements Animal {}
record Cat(String name, boolean indoor) implements Animal {}

@Tool("Registers an animal")
void registerAnimal(Animal animal) { /* dispatched to Dog or Cat */ }

06

返回类型与多模态 / Return Types & Multimodal

EN

void → "Success" sent to LLM. String → returned as-is. Other types → serialized to JSON. For multimodal results, return Image, ImageContent, Content, List<Content>, or Content[]. Supported by Anthropic, Bedrock, and Google AI Gemini.

中文

void → 发送 "Success" 给 LLM。String → 原样返回。其他类型 → 序列化为 JSON。多模态结果可返回 Image、ImageContent、Content、List<Content> 或 Content[]。Anthropic、Bedrock 和 Google AI Gemini 支持。

Java Multimodal Tool Results 
// 返回图片
@Tool("Takes a photo and returns it")
Image takePhoto() {
    byte[] imageBytes = camera.capture();
    return Image.builder()
        .base64Data(Base64.getEncoder().encodeToString(imageBytes))
        .mimeType("image/png")
        .build();
}

// 返回文本 + 图片
@Tool("Takes a photo and returns it with a description")
List<Content> takePhoto() {
    return List.of(
        TextContent.from("Photo taken at " + LocalDateTime.now()),
        ImageContent.from(image)
    );
}
07

AI Service 作为工具 / AI Services as Tools

EN

AI Services can be used as tools for other AI Services — enabling complex agentic systems. A RouterAgent can delegate to specialized experts (Medical, Legal, Technical). Each expert is an AI Service with @Tool-annotated methods. Drawbacks: copy-paste error risk, extra LLM reprocessing cost, and no shared chat memory.

中文

AI Service 可作为其他 AI Service 的工具 —— 实现复杂的 Agent 系统。RouterAgent 可将请求委托给专业专家（医疗、法律、技术）。每个专家都是带有 @Tool 注解方法的 AI Service。缺点：复制粘贴风险、额外的 LLM 处理成本、无共享聊天记忆。

Java Router + Experts 
interface RouterAgent {
    @UserMessage("Analyze the user request and categorize as 'legal', 'medical' or 'technical', then forward to the corresponding expert. Request: '{{it}}'.")
    String askToExpert(String request);
}

interface MedicalExpert {
    @Tool("A medical expert")
    @UserMessage("You are a medical expert. Analyze: {{it}}")
    String medicalRequest(String request);
}

// 配置 RouterAgent 使用三个专家 AI Service 作为工具
RouterAgent routerAgent = AiServices.builder(RouterAgent.class)
    .chatModel(model)
    .tools(medicalExpert, legalExpert, technicalExpert)
    .build();

routerAgent.askToExpert("I broke my leg what should I do");
08

调用参数与上下文 / InvocationParameters & Context

InvocationParameters
传递额外数据
通过 InvocationParameters 向工具传递 LLM 不可见的额外数据（如 userId）。可在 ToolProvider、ErrorHandler、RAG 组件中访问。
InvocationContext
获取调用信息
@Tool 方法可接受 InvocationContext 参数，获取 invocationId、接口名称等调用元信息。
@ToolMemoryId
多用户记忆隔离
将 AI Service 的 @MemoryId 值自动传递到 @Tool 方法中，区分不同用户/会话。
@Description
类与字段描述
使用 @Description 注解类、字段，帮助 LLM 理解参数结构。注意：枚举值上的 @Description 无效。
Java InvocationParameters + @ToolMemoryId 
// InvocationParameters — LLM 不可见
interface Assistant {
    String chat(@UserMessage String msg, InvocationParameters params);
}

class Tools {
    @Tool
    String getWeather(String city, InvocationParameters params) {
        String userId = params.get("userId");
        return weatherService.getWeather(city, getUserPreferences(userId));
    }
}

// @ToolMemoryId — 区分不同用户
interface Assistant {
    String chat(@UserMessage String msg, @MemoryId memoryId);
}

class Tools {
    @Tool
    String addCalendarEvent(CalendarEvent event, @ToolMemoryId memoryId) { ... }
}
09

并发执行与结果访问 / Concurrent Execution & Access

EN

Concurrent execution: By default, tools execute sequentially. Enable executeToolsConcurrently() or executeToolsConcurrently(Executor) for parallel execution. With ChatModel, single tool runs in caller thread; multiple tools use Executor. With StreamingChatModel, all tools use Executor threads.

Accessing results: Wrap return type in Result<T> to get toolExecutions() list. In streaming, use onToolExecuted callback.

中文

并发执行：默认顺序执行。启用 executeToolsConcurrently() 或 executeToolsConcurrently(Executor) 实现并行。ChatModel 下单个工具在调用线程执行，StreamingChatModel 下所有工具使用 Executor 线程。

访问结果：将返回类型包装为 Result<T> 获取 toolExecutions() 列表。流式模式下使用 onToolExecuted 回调。

Java Result + onToolExecuted 
// 同步模式
interface Assistant {
    Result<String> chat(String userMessage);
}

Result<String> result = assistant.chat("Cancel my booking 123-456");
List<ToolExecution> toolExecs = result.toolExecutions();
ToolExecution exec = toolExecs.get(0);
String reqResult = exec.result();           // 文本结果
List<Content> contents = exec.resultContents();  // 含图片
Object obj = exec.resultObject();               // 原始返回值

// 流式模式
tokenStream
    .onToolExecuted(toolExecution -> System.out.println(toolExecution))
    .onPartialResponse(...)
    .onCompleteResponse(...)
    .start();
10

编程式与动态工具 / Programmatic & Dynamic Tools

EN

Programmatic: Define tools via ToolSpecification + ToolExecutor pairs, wrapped in AiServiceTool. Tools can be loaded from databases or config files.

Dynamic (ToolProvider): Tools provided per invocation based on ToolProviderRequest (user message, memory ID, invocation parameters). Static and dynamic tools are merged together.

中文

编程式：通过 ToolSpecification + ToolExecutor 对定义工具，包装在 AiServiceTool 中。工具可从数据库或配置文件加载。

动态（ToolProvider）：每次调用时根据 ToolProviderRequest（用户消息、memory ID、调用参数）动态提供工具。静态和动态工具会合并。

Java Programmatic + Dynamic 
// 编程式工具
AiServiceTool tool = AiServiceTool.builder()
    .toolSpecification(toolSpecification)
    .toolExecutor(toolExecutor)
    .build();

Assistant assistant = AiServices.builder(Assistant.class)
    .chatModel(chatModel)
    .tools(List.of(tool))
    .build();

// 动态工具 — 仅当消息含 "booking" 时提供
ToolProvider toolProvider = (toolProviderRequest) -> {
    if (toolProviderRequest.userMessage().singleText().contains("booking")) {
        return ToolProviderResult.builder()
            .add(toolSpecification, toolExecutor)
            .build();
    }
    return null;
};
11

工具搜索 / Tool Search

EN

When you have many tools (dozens/hundreds), sending all on every request wastes tokens. Tool Search lets the LLM discover tools dynamically: initially only tool-search tools are exposed, the LLM calls them to find relevant tools, and matching tools are added to subsequent requests. Found tools are accumulated (merged, not replaced). Two built-in strategies: SimpleToolSearchStrategy (keyword) and VectorToolSearchStrategy (embeddings).

中文

当工具数量众多（数十/数百）时，每次请求都发送全部工具会浪费 token。工具搜索让 LLM 动态发现工具：初始仅暴露工具搜索工具，LLM 调用它们查找相关工具，匹配的工具被添加到后续请求中。已找到的工具会累积（合并而非替换）。两种内置策略：SimpleToolSearchStrategy（关键词）和 VectorToolSearchStrategy（嵌入向量）。

1
初始请求
LLM 仅看到工具搜索工具
2
工具搜索
LLM 调用搜索工具，描述所需工具
3
工具暴露
匹配的工具添加到下一请求
4
工具执行
LLM 正常调用找到的工具
Java ToolSearch Configuration 
Assistant assistant = AiServices.builder(Assistant.class)
    .chatModel(chatModel)
    .chatMemory(chatMemory)
    .tools(tools)
    .toolProvider(mcpToolProvider)
    .toolSearchStrategy(new SimpleToolSearchStrategy())
    .build();

// 始终可见的工具
@Tool(searchBehavior = ALWAYS_VISIBLE)
String getWeather(String city) { ... }
12

即时返回 / ReturnBehavior

EN

TO_LLM (default): Tool result sent back to LLM for reprocessing.
IMMEDIATE: Tool result returned directly, skipping LLM reprocessing. Result.content() will be null; use result.toolExecutions().
IMMEDIATE_IF_LAST: Returns immediately when this tool is the last in the LLM's response — saves one LLM round trip for multi-step action sequences.

中文

TO_LLM（默认）：工具结果发回 LLM 重新处理。
IMMEDIATE：工具结果直接返回，跳过 LLM 重新处理。Result.content() 为 null；使用 result.toolExecutions() 获取。
IMMEDIATE_IF_LAST：当此工具是 LLM 响应中的最后一个时立即返回 —— 为多步骤操作序列节省一次 LLM 往返。

Tool calls in responseOutcomeWhy
[IMMEDIATE]return immediatelyonly tool is IMMEDIATE
[IMMEDIATE, IMMEDIATE]return immediatelyevery tool is IMMEDIATE
[IMMEDIATE, TO_LLM]reprocessTO_LLM disqualifies
[IMMEDIATE_IF_LAST]return immediatelylast tool is IMMEDIATE_IF_LAST
[TO_LLM, IMMEDIATE_IF_LAST]return immediatelylast tool is IMMEDIATE_IF_LAST
[IMMEDIATE_IF_LAST, TO_LLM]reprocessnot last + TO_LLM present
[IMMEDIATE_IF_LAST(err)]reprocessany error disables immediate return
13

错误处理 / Error Handling

EN

Hallucinated Tool Names: Configure hallucinatedToolNameStrategy to return a helpful error message instead of throwing.
Tool Arguments Errors: Use ToolArgumentsErrorHandler — return error text for LLM to self-correct, or throw to abort.
Tool Execution Errors: Use ToolExecutionErrorHandler — default sends exception message to LLM (not recommended for production). Sanitize with a custom handler.

中文

幻觉工具名：配置 hallucinatedToolNameStrategy 返回有用错误信息而非抛出异常。
工具参数错误：使用 ToolArgumentsErrorHandler —— 返回错误文本让 LLM 自我纠正，或抛异常中止。
工具执行错误：使用 ToolExecutionErrorHandler —— 默认发送异常消息给 LLM（生产环境不推荐），用自定义处理器清理。

Java Error Handlers 
// 幻觉工具名处理
Assistant assistant = AiServices.builder(Assistant.class)
    .chatModel(model)
    .tools(new HelloWorld())
    .hallucinatedToolNameStrategy(toolExecutionRequest ->
        ToolExecutionResultMessage.from(toolExecutionRequest,
            "Error: there is no tool called " + toolExecutionRequest.name()))
    .build();

// 参数错误 — 推荐：让 LLM 重试
.toolArgumentsErrorHandler((error, ctx) ->
    ToolErrorHandlerResult.text(error.getMessage()))

// 执行错误 — 生产环境推荐：返回通用消息
.toolExecutionErrorHandler((error, ctx) ->
    ToolErrorHandlerResult.text("Tool execution failed."))
安全警告：默认的工具执行错误处理会将原始异常消息发送给 LLM。生产环境中这可能泄露内部数据（堆栈跟踪、文件路径、凭证、PII 等）。请配置自定义 ToolExecutionErrorHandler 返回通用或经过清理的错误描述。
14

模型上下文协议 / MCP

EN

You can also import tools from MCP (Model Context Protocol) servers. This allows LangChain4j applications to leverage tools exposed by MCP-compatible services, expanding the tool ecosystem beyond locally defined methods.

中文

你也可以从 MCP（模型上下文协议）服务器导入工具。这使得 LangChain4j 应用程序能够利用 MCP 兼容服务暴露的工具，将工具生态扩展到本地定义的方法之外。

总结：Tools (Function Calling) 是 LangChain4j 中最强大的功能之一。从低级的 ToolSpecification 到高级的 @Tool 注解，从静态工具到动态 ToolProvider，从工具搜索到即时返回，LangChain4j 提供了完整的工具调用生命周期管理。结合 MCP 协议，你可以构建真正可扩展的 AI Agent 系统。
